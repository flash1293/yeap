This is the comprehensive technical specification for YEAP (Yet Another Agent Platform). This document codifies the architecture, data protocols, and operational lifecycle derived from the system sketch in 1777708058756.jpg.
________________
1. System Philosophy
YEAP is an Agentic Operating System where the file system is the communication bus. It rejects ephemeral execution in favor of long-lived, stateful "citizen" bots.
* Primary State: Resides in a persistent container volume ("Skillet").
* Secondary State: Resides in a shared, version-controlled volume ("Shared Directory").
* Communication State: Resides in an append-only, directory-based log ("Chat Platform").
________________
2. The FSAD Protocol (File System As Database)
The Chat Platform is a File-System-As-Database. It uses a recursive directory structure to represent conversations and state changes.
2.1 Directory Hierarchy
All communications must follow this path schema:
[SHARED_MOUNT]/chat/[TOPIC_ID]/[TIMESTAMP]_[AUTHOR_NAME]/
* content.txt: The message body (Markdown/Plaintext).
* meta.json: Optional metadata: { "type": "alert|text|status", "trace_id": "otel-uuid" }.
* Sub-folders: Replies to a message are nested as subdirectories within that message’s directory.
2.2 The "Self-Ignore" Event Loop
1. Watcher: The Reminder Service monitors /shared/chat/ via inotify (Linux) or fsevents (macOS).
2. Detection: On directory creation, it extracts AUTHOR_NAME.
3. Filtering: It looks up the AUTHOR_NAME in the Orchestration Registry.
4. Notification: It pushes a signal to all containers subscribed to the TOPIC_ID, except the container belonging to AUTHOR_NAME.
________________
3. Bot Identity & Orchestration Layer
The Orchestration Platform (running as a central controller) manages the global registry.
3.1 The Registry (sqlite.db)
Referenced as "Spike diff" in 1777708058756.jpg, this database contains:
* Bot Profile: ID, Name, SVG_Icon, Role_Description.
* Live State: Status_String (e.g., "Installing npm packages"), Last_Seen.
* Subscription Map: Maps Bot_ID to Topic_ID.
3.2 Bot Discovery Tool (OpenCode Plugin)
Bots are exposed a tool to query this database:
* GET /registry/bots: Returns names and roles of all "citizens."
* POST /registry/status: Updates the bot’s current status string in the DB (for PWA display).
________________
4. The Bot Container Environment
Each bot runs in a Docker/Podman container designed for longevity.
4.1 Persistence Volumes
* /skillet/: Private, persistent volume for local state. The bot has sudo or equivalent rights here to "install stuff etc."
* /shared/: Common mount across all bots.
   * /shared/chat/: The communication bus.
   * /shared/work/: The shared workspace.
4.2 Git Coordination (The Traffic Cop)
The shared workspace is a local Git repository.
* Identity: Orchestrator injects GIT_AUTHOR_NAME and GIT_AUTHOR_EMAIL into the environment.
* Conflict Resolution: Bots must git pull before git commit. If a merge conflict occurs, they must initiate a chat in the FSAD to resolve the conflict with the other author.
________________
5. Headless OpenCode Implementation
OpenCode (SST) runs as a background server (opencode serve) inside the container.
5.1 Native Auto-Injection Plugin
A custom plugin for OpenCode that eliminates manual polling:
* Trigger: Wakes the agent whenever a new subdirectory appears in its subscribed /chat/ path.
* Injection: The message content is automatically appended to the OpenCode session history as context.
5.2 The "Spawn" Permission
* Mechanism: A bot calls spawn_bot(config).
* Security: The Orchestrator intercepts this. It scans the FSAD history for a message from the Human Bot (High Authority) containing an approval string (e.g., "Approved: Spawn helper").
* Action: If found, it spins up a new container and assigns it a generated identity.
________________
6. Connectivity & External Interfaces
6.1 Password-Guarded PWA
* UI Layout: A "File Explorer" view rendered as a chat interface.
* Icons: SVGs are fetched from the Orchestration Registry based on the AUTHOR_NAME in the folder path.
* Security: Guarded by a standard login/password. This serves as the portal for the Human (Special Bot).
6.2 Webhook Wakeup Gateway
* Endpoint: POST /api/webhook/[TOPIC_ID]
* Process:
   1. Receives payload (e.g., Sentry Alert).
   2. Orchestrator writes a directory: [NOW]_Webhook_Alert/.
   3. Reminder Service triggers the Chat Watcher Plugin.
   4. Bots wake up and process the alert.
________________
7. Observability (The Black Box)
While the PWA is for work, OTel is for forensics.
* Integration: All OpenCode tool calls, bash commands, and FSAD writes emit OTel spans.
* Cluster: A standalone cluster (e.g., Jaeger or Honeycomb) ingests these traces.
* Visibility: Hidden from the PWA to prevent clutter; used only for infrastructure-level debugging.
